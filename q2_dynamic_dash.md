# HubAgent Feature Brief: Q2 Revenue Operating Dashboard / Goal Tracker v2

## Overview

This feature is the next evolution of the existing Q2 Goal Tracker inside HubAgent.

The current Q2 Goal Tracker is mainly a historical planning and benchmarking tool. It uses Q1 data to establish baseline assumptions for Q2, such as conversion rates, deal progression, speed to demo, speed to close, and the throughput required to hit quarterly goals if historical patterns remain roughly the same.

The new feature should transform that concept into a much more comprehensive, live revenue operating dashboard inside HubAgent.

This new module should not just look backward. It should combine historical baselines, real-time HubSpot data, engagement history, selective LLM analysis, and intelligent forecasting to help answer whether the business is truly on track to hit the Q2 ARR target, where performance is breaking down, which initiatives are working, and which deals are actually likely to close.

This should feel like a real operating system for the quarter, not a static report.

---

## Core Purpose

The purpose of this feature is to create a live, intelligent Q2 revenue operating dashboard that helps leadership understand:

- whether the team is on pace to hit Q2 ARR goals
- whether enough leads are being generated by source and by week
- whether enough deals are being created from those leads
- whether those deals are being worked properly by account executives
- whether strategic initiatives are producing the expected pipeline contribution
- whether the current pipeline is truly healthy or only appears healthy on the surface
- what the likely quarter outcome is based on actual engagement and momentum inside deals

This system should act as both a measurement layer and a decision-support layer.

---

## High-Level Product Thesis

The current Q2 tracker establishes what would need to happen if historical patterns remain true.

This new module should answer what is actually happening now.

It should compare:
- baseline expectations from Q1
- required pacing for Q2 goals
- current-quarter live performance
- AI-assisted interpretation of real deal health

The result should be a dashboard that tells the user:
- where we are on track
- where we are off track
- why we are off track
- which areas need follow-up
- what the current likely forecast is if present trends continue

---

## Important Design Principle

This feature should not use AI for everything.

A large portion of this module should be based on:
- HubSpot metadata
- object properties
- deterministic logic
- rule-based calculations
- historical conversion math
- timeline/event sequencing

LLM processing should be used selectively only where it adds meaningful value, such as:
- interpreting deal engagement quality
- analyzing email tone and buyer sentiment
- reviewing notes and context
- making an independent likelihood-to-close judgment
- summarizing why a deal is at risk or strong

The system should be calculation-first and AI-assisted, not AI-first in every component.

## What Already Exists Conceptually

There is already an existing Q2 Goal Tracker that uses Q1 performance as the planning baseline.

There is also already a pattern in the codebase for building or interpreting a deal timeline of events. That existing pattern should be found and reused where appropriate rather than reinvented.

The coding agent should actively search the codebase for:
- the existing Q2 Goal Tracker logic and UI patterns
- any existing deal timeline/event reconstruction patterns
- any existing engagement analysis flows
- any existing deep-context activator pattern or contextual prompt-building pattern
- any existing forecasting or dashboard card patterns
- any reusable HubSpot sync/data processing logic already present in HubAgent

The goal is not to bolt on a disconnected feature. The goal is to evolve the existing application intelligently and reuse established patterns wherever possible.

## What This Feature Must Be Able To Do

## 1. Historical Baseline and Goal Context

The feature should understand the historical Q1 baseline and use it as the context for Q2 planning and pacing.

It should be able to reference historical metrics such as:
- lead-to-deal behavior
- deal-to-demo progression
- demo completed to closed won rate
- sales cycle timing
- throughput required to hit quarterly targets
- any other existing baseline measures already present in the current Q2 tracker

This baseline is important because it provides the "required pace" framework against which live Q2 performance will be evaluated.

---

## 2. Lead Creation and Channel Pacing

The system should understand how many leads must be created by source/channel over a given time period in order to support the Q2 ARR goal.

It should use HubSpot data and historical funnel assumptions to determine:
- required lead volume by source
- actual lead volume by source
- whether lead creation is pacing correctly
- whether each channel is ahead of pace, on pace, or behind pace
- how weekly lead generation is tracking against expectations

This should make it possible to monitor, in real time, how marketing and other lead-generating initiatives are performing relative to the quarter goal.

This is not just raw reporting. It should connect top-of-funnel activity to the actual Q2 ARR objective.

---

## 3. Deal Creation and Pipeline Throughput Tracking

The system should track whether enough deals are being created from leads, and whether the volume and pace of deal creation are sufficient relative to the quarter goal.

It should help answer:
- are enough deals being created overall
- are enough deals being created from each source
- are those deals arriving at the right pace
- is current pipeline generation sufficient given the Q2 target
- how current creation patterns compare with historical conversion expectations

This turns the dashboard into a live pipeline throughput monitor, not just a lead tracker.

---

## 4. Weekly and Real-Time Operating Visibility

The dashboard should make it possible to understand week by week how the quarter is progressing.

The intent is to catch issues early, not only report them later.

It should surface whether:
- weekly lead production is sufficient
- weekly deal creation is sufficient
- current pacing suggests underperformance
- certain channels or initiatives are falling short
- intervention is needed before the quarter drifts too far off target

This should feel operational, current, and alive.

---

## 5. Strategic Initiative Tracking

The system must also track new or strategic lead-source initiatives that are important to leadership.

Examples specifically mentioned:
- CEO-led channel partner initiative
- co-destiny referral initiative
- newly created lead sources added for tracking

For these initiatives, the system should support defining expectations in advance, such as:
- what volume of leads should be produced
- what deal creation contribution should be expected
- what pace is needed by week or by period
- what on-track versus off-track looks like

The system should then monitor actuals against those expectations in real time so follow-up and accountability are possible.

This is important because these initiatives are not generic sources. They are strategic bets that need explicit visibility.

---

## 6. Deal-Level Timeline Reconstruction

For each deal, the system should be able to build or interpret a timeline of events.

This should rely primarily on metadata and structured engagement history.

It should evaluate things like:
- deal creation
- major engagement milestones
- calls
- emails
- notes
- meetings
- stage movement
- response timing
- recency of activity
- cadence of follow-up
- gaps or stalls in activity

The coding agent should search the codebase for any already-implemented pattern that constructs or interprets a timeline of deal events and reuse that pattern.

This timeline is foundational because multiple downstream judgments will depend on it.

---

## 7. AE Execution Review Against Q2 Process Changes

A major purpose of this feature is not just to track whether deals exist, but whether they are being worked effectively by account executives.

The system should evaluate each deal to determine whether AE execution appears aligned with the process improvements that were introduced for Q2.

This should include both:
- structured analysis of activity and timing
- AI-assisted interpretation of communications and notes

The goal is to judge whether the AE appears to be handling the deal with the expected rigor and discipline.

This can become a coaching and process-enforcement layer, not only a forecasting layer.

It should help answer:
- is the rep following the intended process
- is follow-up happening with enough consistency
- is the deal being worked proactively
- does engagement reflect effective execution
- does the deal timeline suggest good handling or weak handling

---

## 8. AI-Assisted Deal Interpretation

For deal-level analysis, the system should selectively use LLM processing to evaluate unstructured context such as:
- emails exchanged with the prospect
- notes on the deal
- communication style and tone
- customer responses
- whether the buyer appears engaged, hesitant, delayed, positive, or drifting
- whether the conversation suggests real momentum or superficial activity

The system should combine:
- metadata and structured timeline signals
- communication analysis
- contextual notes
- overall engagement quality

This should produce a richer, more independent judgment than stage alone.

---

## 9. Independent Likelihood-to-Close Assessment

The system should generate an independent view of whether each deal is likely to close in the current quarter.

This should not rely only on stage or owner opinion.

Instead, it should assess likelihood to close based on:
- engagement activity
- timeline progression
- response behavior
- communication quality
- email tone and sentiment
- cadence of meetings and follow-up
- notes/context
- whether the buyer appears to be moving meaningfully toward purchase

Possible output states may conceptually include:
- likely to close this quarter
- unlikely to close this quarter
- at risk
- poor sentiment
- insufficient information to assess

The system should also provide a rationale or explanation so the output does not feel like a black box.

The exact labels can be designed intelligently, but the underlying need is clear: the system must form a real opinion on the true health and likely outcome of each deal.

---

## 10. Real-Time Rolling Forecast

Once individual deals have been analyzed, the system should roll those judgments up into a live, dynamic forecast for the quarter.

This should go beyond simple pipeline total or stage-based forecasting.

It should account for:
- likely deals
- unlikely deals
- at-risk deals
- deals with insufficient signal
- any human-adjusted judgments

This should produce a rolling point of view on the quarter, updated as deal conditions evolve.

The user should be able to look at the dashboard and understand:
- what the quarter currently appears likely to produce
- how that compares to goal
- how much of the pipeline appears truly healthy
- where the risk sits

This forecast layer is one of the highest-value parts of the feature.

---

## 11. Human Override Capability

The system must support manual override of AI-based deal judgments.

This is important because there will be situations where:
- the system says unlikely, but leadership has knowledge the model does not
- the system says likely, but a human believes the signal is misleading
- the system says not enough information, but the internal team knows the deal is real
- leadership wants to adjust the deal's effective forecast weight

This override should not be an afterthought. It is a core requirement.

The user should be able to:
- override the system's judgment
- see what the original judgment was
- understand that a manual adjustment has been made
- preserve trust in the system while still allowing leadership control

Auditability and transparency of overrides would be beneficial.

---

## 12. Intelligent Summary and Decision Support

The dashboard should not only display numbers. It should help the user understand what matters.

It should ideally surface insights like:
- which channels are behind pace
- which initiatives are underperforming
- which deals are most at risk
- which reps may not be executing the process effectively
- whether marketing output is sufficient
- whether the pipeline appears stronger or weaker than stage-based reporting would suggest
- what seems to be driving the current quarter outlook

The experience should feel like an intelligent operator's dashboard, not a collection of disconnected reports.

---

This needs to feel like a polished, thoughtfully designed core module inside HubAgent.

It should feel executive-ready, but also operationally useful.

The interface should not feel cluttered or like a traditional dense CRM report. It should feel modern, high-signal, and deliberate.

The experience should communicate:
- confidence
- clarity
- intelligence
- urgency where appropriate
- focus on what matters most

The final design should be clean and premium, with a modern SaaS product feel.

## Wireframe in Words: Final-State Experience

The coding agent should first think through and describe the final user experience before planning the code changes.

The UI should likely feel like this in its final state:

### Entry / Landing Experience

When I open this feature inside HubAgent, I should feel like I am entering the command center for the quarter.

At the top, there should be a strong summary region that immediately answers the most important questions:
- Q2 ARR target
- current projected outcome
- amount needed to hit goal
- whether we are on track, behind, or at risk
- a visual or textual confidence signal about the forecast

This top area should feel crisp and decisive, not noisy.

### Executive Summary Band

Near the top, there should be a set of high-signal summary cards or panels that make the quarter legible immediately.

Examples of what I should be able to understand at a glance:
- target vs projected ARR
- required pace vs actual pace
- leads created vs required leads
- deals created vs required deals
- likely-to-close weighted outcome
- number of at-risk deals
- number of initiatives on track vs off track

These cards should feel sharp, elegant, and easy to scan.

### Main Dashboard Sections

Below the summary, the dashboard should likely break into several major zones:

#### 1. Goal Pacing / Funnel Health
A section that shows whether lead generation and deal creation are keeping pace with what is required to hit Q2 goals.

This should likely be visually clear and easy to compare:
- required vs actual by week
- required vs actual by source
- channel contribution pacing
- trend direction

This area should make it easy to spot underperformance early.

#### 2. Strategic Initiative Tracker
A section specifically for named initiatives such as CEO-led channel partner efforts and co-destiny referrals.

This should show:
- expectation
- actual progress
- pacing status
- whether follow-up is needed

These initiatives should feel visible and accountable, not buried in generic source data.

#### 3. Deal Intelligence / Execution Zone
A section that lets me review the health of individual deals.

This should likely support a list or table view with high-signal columns and strong filtering, where I can quickly see:
- deal name
- owner
- source
- stage
- recent activity
- AI likelihood to close
- sentiment
- risk level
- whether the system lacks enough information
- whether a human override exists

Clicking into a deal should reveal a richer detail view.

#### 4. Deal Detail Experience
When I open an individual deal, I should see a well-organized detail experience that feels like an intelligence layer on top of HubSpot.

This detail experience should likely include:
- concise deal summary
- timeline of events
- key engagement history
- AI interpretation of momentum
- sentiment and close-likelihood rationale
- execution assessment relative to Q2 process expectations
- any warning signals
- any human override that has been applied

This should feel like a readable operating brief, not a messy data dump.

#### 5. Forecast View
A section that explains how the current rolling forecast is being formed.

This should help me understand:
- what is likely contributing to the forecast
- what is weak
- how much pipeline is healthy vs questionable
- what the system believes is realistically closeable this quarter

This should feel useful for leadership, not abstract.

The visual design should be modern and refined.

The experience should likely use:
- a clean layout

- operational

- not cluttered

Think more "mission control for revenue operations" and less "busy analytics dashboard."
