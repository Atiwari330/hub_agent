Opus Customer Authorization and Approval Policy v1
Purpose
This policy defines who is authorized to request certain actions from Opus Support, when written approval is required, how Tier 1 Support should validate that approval, and what actions must not be taken without proper customer authorization.
The purpose of this policy is to:
protect customer accounts and data
reduce unauthorized changes
protect against phishing or impersonation
preserve workflow and configuration integrity
ensure support actions are taken only with appropriate customer authority
This policy should be used together with the Opus Support Authority Matrix.

Core Policy Principle
Opus Support may only perform certain customer-impacting actions when those actions are requested or approved by a properly authorized customer contact.
Tier 1 Support should never assume that any employee of a customer organization is automatically authorized to request:
access changes
user changes
permissions changes
template or form changes
report requests
data exports
workflow or configuration changes
scoped services work
When authorization is required, it must come from the correct customer authority.

Default Support Philosophy
Tier 1 Support should default to being:
helpful
educational
professional
cautious
process-driven
When a request requires authorization, the goal is not to be difficult. The goal is to make sure Opus is acting only on valid requests from the right customer representatives.
Whenever practical, Support should guide the customer on how to complete administrative actions themselves rather than taking over those actions directly.

Definitions
Authorized Account POC
The primary customer contact or designated operational decision-maker on file for the account who is recognized by Opus as having authority to approve account-impacting requests.
Authorized Admin
A designated customer representative on file who has authority to request or approve specific administrative actions within the customer’s environment.
End User
A staff member, clinician, biller, scheduler, or other user at the customer organization who uses the platform but is not necessarily authorized to approve account-level actions.
Written Approval
Written confirmation from an authorized account POC or authorized admin on file. This approval must be clear enough for Support to reasonably rely on it before acting.
Customer Authorization Required
A request category where Tier 1 Support may not proceed unless the request is approved by an authorized customer contact.
Internal Approval Required
A request category where Opus internal review or approval is also needed in addition to or separate from customer approval.

General Authorization Rule
If a request changes access, permissions, configuration, templates, reports, exports, or other material aspects of the customer environment, Tier 1 should assume authorization is required unless the Authority Matrix clearly states otherwise.
If the authority of the requester is unclear, Tier 1 should pause and verify before acting.

Categories of Requests That Commonly Require Authorization
The following types of requests generally require authorization from an authorized customer contact:
disabling or deactivating a user
modifying user roles or permissions
simple template or form changes made by Opus on the customer’s behalf
deleting forms or templates
any workflow-affecting change
report requests
mass data export requests
complex form builds
scoped services requests
advanced training requests
vendor-side settings changes initiated on behalf of the customer
billing or operational configuration changes
The exact authority rules for each request type should be confirmed in the Authority Matrix.

Requests That Should Not Be Accepted from a Regular End User Alone
Tier 1 Support should not act on the following requests when they come only from a regular end user without proper approval:
create a new user account
grant a new user access
disable another user
modify another user’s permissions
change provider credentials
change billing configuration
change ePrescribe settings
delete forms or templates
request a new report as if it were routine support work
request a mass export of data
request significant workflow changes
request custom project-type work
Support may still respond helpfully, but should not execute the action without proper authorization.

Acceptable Forms of Written Approval
Written approval may include:
an email from the authorized account POC or authorized admin on file
a ticket reply from the authorized account POC or authorized admin on file
a clearly documented written request in an approved customer communication channel that Opus formally relies upon
The approval should clearly identify:
what is being requested
who is approving it
enough detail for Support to understand what action is authorized
If the request is ambiguous, Tier 1 should ask clarifying questions before acting.

Unacceptable or Insufficient Approval
The following should generally not be treated as sufficient approval by themselves:
a verbal statement with no written confirmation
an email from someone whose authority is unknown
a forwarded message with unclear origin
“someone at our office told me to ask”
“I usually handle this”
implied approval
historical habit without current confirmation
a request from a regular end user for changes affecting someone else’s access or the overall environment
If there is uncertainty, Tier 1 should not guess.

How Tier 1 Should Verify Authorization
When a request requires authorization, Tier 1 should:
Review the account in the relevant internal system of record.
Confirm who is listed as the authorized account POC or authorized admin.
Confirm whether the current requester matches that authorized contact.
Confirm that the written approval is clear enough to rely on.
If the request is unclear or the sender is not clearly authorized, seek confirmation before acting.
Document the approval source in the ticket.
The default expectation is that authorization is validated before action is taken, not after.

Documentation Requirement
Whenever Tier 1 acts based on customer approval, the support record should clearly reflect:
who approved the request
when the approval was received
what action was approved
where the approval was found
what action Support took
This creates traceability and protects both Opus and the customer.

When Tier 1 Should Pause and Escalate
Tier 1 should pause and escalate to the Customer Support Manager when:
the authority of the requester is unclear
two customer contacts appear to conflict
the request seems unusually risky
the request could create significant workflow disruption
the request relates to security or possible unauthorized access
the requester is pressuring Support to bypass policy
the customer disputes who is authorized
the change is unusually broad or sensitive
the rep is unsure whether the request is support work or scoped work
When in doubt, escalation is safer than improvisation.

Special Rule for User Creation and New Access
Tier 1 Support should not create new user accounts or grant new access based on an end-user request.
These requests should be redirected to the authorized customer admin or account POC according to established account management rules.
This is a strict safeguard against:
phishing
impersonation
unauthorized system access
poor access hygiene
Even if the requester appears legitimate, Tier 1 should follow policy.

Special Rule for User Disablement
Tier 1 may disable or deactivate a user only when:
the request is in writing, and
it comes from the authorized account POC or authorized admin on file
If there is a documented emergency or security escalation path, that path may override the standard flow as defined by internal policy.
If not, Tier 1 should not disable a user based on a casual or informal request.

Special Rule for Permissions and Role Changes
Tier 1 may modify existing user permissions or roles only when written approval exists from the authorized account POC or authorized admin on file.
Even when this is allowed, the preferred first approach is to teach the customer how to make the change themselves when appropriate.

Special Rule for Templates, Forms, and Similar Changes
Tier 1 may perform simple template or form changes only with written approval from the authorized account POC or authorized admin on file.
Support should not make structural or workflow-impacting changes based solely on an end-user request.
Complex or large-volume changes may need to be routed into a scoped work or SOW process instead of being handled as routine support.

Special Rule for Reports, Exports, and Scoped Work
Requests involving reports, mass exports, complex forms, advanced training, or similar project-type work should not be treated as ordinary ticket work simply because a customer asked for them.
Even when a customer contact is authorized to request them, these items often require:
clarified scope
internal review
pricing review
SOW creation
defined timeline
formal sign-off
Tier 1 should gather information and route appropriately.

Handling Requests from Non-Authorized End Users
When a regular end user asks Support to do something they are not authorized to approve, Tier 1 should remain helpful and professional.
Recommended posture:
explain that the requested action requires approval from the authorized customer contact
identify the correct approval path
offer to provide guidance or instructions where appropriate
avoid sounding rigid or dismissive
document the interaction clearly
The goal is to maintain good customer experience while still protecting the account.

Recommended Customer-Facing Language
Example: Access Creation Request
“Thanks for reaching out. For security reasons, our team is not able to create new user accounts or grant new access based on an end-user request alone. The request will need to come from your organization’s authorized account contact or designated admin on file. I’m happy to help point you in the right direction.”
Example: Permission Change Request
“Happy to help. Changes to user roles or permissions require written approval from your authorized account contact or admin on file. If you’d like, I can help explain the process or review the request once that approval is provided.”
Example: Form Change Request
“I’m happy to help with this. Because this change affects your organization’s forms/workflows, we’ll need written approval from your authorized account contact or admin on file before we make the update on your behalf.”
Example: Unclear Authority
“Thanks for the request. Before we proceed, I need to confirm authorization for this change with the appropriate account contact on file. Once that’s confirmed, we can move forward.”

Internal Documentation Standard
For any authorization-dependent action, Tier 1 should include a note in the ticket that captures:
request type
whether authorization was required
who provided the approval
how it was verified
whether the action was taken or escalated
any unusual circumstances
This should be done consistently.

Policy Exceptions
Exceptions should be rare.
If a rep believes an exception is needed, the matter should be escalated to the Customer Support Manager or other designated internal leader before action is taken.
Support reps should not create their own exceptions in the moment.

Practical Examples
Example 1: Clinician Asks for Another Clinician to Be Disabled
Do not act based on that request alone. Confirm written approval from the authorized account POC or authorized admin.
Example 2: Office Manager on File Emails Asking to Disable a User
If the office manager is an authorized admin on file and the request is clear, Tier 1 may proceed according to the Authority Matrix.
Example 3: Biller Wants Their Own Role Changed
Tier 1 should not make the change based solely on that request unless written approval from the authorized account contact exists.
Example 4: End User Wants a Form Deleted
Do not delete the form based on the request alone. Confirm written authorization from the proper customer contact.
Example 5: Unknown Sender Emails Asking for Access
Do not act. Treat as unauthorized unless properly validated through the established authorization path.

Final Operating Statement
Authorization is not a formality. It is a core control.
Opus Support should be easy to work with, but not careless.
When a request affects access, permissions, workflow, configuration, or customer environment integrity, Tier 1 must make sure the request is coming from the right person with the right authority before acting.
When in doubt, verify first and escalate if needed.
