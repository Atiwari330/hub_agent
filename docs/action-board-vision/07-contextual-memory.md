# Phase 7: Contextual Memory & Running Narrative

## Goal

Give the analysis engine memory of its previous analyses so it can produce **diff-aware updates** instead of starting from scratch every time. Agents see not just "here's what's happening now" but "here's what changed since last time" — a running narrative of how the ticket evolved.

## Why This Phase Is Last

It builds on the multi-pass architecture (Phase 2) and benefits from all the event-driven data (Phase 3). It's also a refinement — the system works well without it. This phase makes it work *beautifully*.

---

## Current State

- Each analysis overwrites the previous one entirely (`upsert` on `hubspot_ticket_id`)
- No record of what the previous analysis said
- No way to see how a ticket's situation, temperature, or action items evolved
- The LLM starts fresh every time — it doesn't know what it said last time

This means:
- The summary might flip-flop in tone between analyses
- New action items might repeat things that were already tried
- The agent has no sense of trajectory ("Is this getting better or worse?")

---

## Target State

### 1. Analysis History

Every analysis pass result is preserved (already set up in Phase 2's `analysis_pass_results` table). This phase uses that history.

### 2. Diff-Aware Analysis

Each pass receives its previous output and is asked to produce an **update**, not a fresh analysis:

**Situation Pass with memory:**
```
PREVIOUS SITUATION SUMMARY (from 2 hours ago):
"ABC Corp reported a sync error affecting their billing module. Engineering created LINEAR-4521
to investigate. Customer was told to expect an update within 24 hours."

WHAT'S CHANGED SINCE THEN:
- Customer sent 1 new message (2 hours ago): asking for update
- Agent responded (30 min ago): provided interim status
- LINEAR-4521 state changed: In Progress → In Review

YOUR JOB: Update the situation summary to reflect what changed.
Focus on what's NEW — don't repeat unchanged context.
```

**Action Item Pass with memory:**
```
PREVIOUS ACTION ITEMS (from 2 hours ago):
1. [COMPLETED] Reply to customer with engineering update (agent responded 30 min ago)
2. [ACTIVE] Monitor LINEAR-4521 for resolution
3. [ACTIVE] Schedule follow-up if not resolved by EOD Friday

WHAT'S CHANGED:
- Agent responded to customer (item 1 addressed)
- Linear issue moved to "In Review" (item 2 still relevant but context changed)

YOUR JOB: Update the action list.
- Keep items that are still relevant
- Mark items that were addressed
- Add new items based on changes
- DON'T regenerate from scratch — evolve the existing list
```

### 3. Ticket Evolution Timeline

A new UI section showing the analysis narrative over time:

```
TICKET TIMELINE (auto-generated):

Mar 24, 2:00 PM — Customer replied asking for ETA on billing fix
  → Temperature shifted: calm → frustrated
  → New action: Provide specific timeline from LINEAR-4521

Mar 24, 2:30 PM — Agent responded with interim update
  → Action completed: Reply to customer ✓
  → Temperature held: frustrated (customer acknowledged but wants resolution)

Mar 24, 3:15 PM — LINEAR-4521 moved to "In Review"
  → Updated: Engineering fix is in code review
  → New action: Notify customer when fix is deployed

Mar 24, 5:00 PM — Fix deployed to staging
  → Updated: Engineering confirmed fix works in staging
  → New action: Ask customer to verify fix in production tomorrow AM
```

This timeline is auto-generated by comparing consecutive analyses — what changed in the situation, temperature, and action items.

### Step 2: Memory-Aware Passes

**Modified files:** Each pass in `src/lib/ai/passes/`

Each pass receives a `changes` parameter alongside its regular context. When `changes.isFirstAnalysis` is true, it runs normally. When it has previous context, it runs in "update mode":

The pass prompt structure becomes:
```
[If first analysis]
  "Analyze this ticket for the first time..."

[If update]
  "Here is your previous analysis from [X hours ago]:
   [previous output]

   Since then, the following changed:
   [changes summary]

   Update your analysis to reflect these changes.
   Preserve continuity — don't contradict your previous assessment without reason.
   If your assessment changed, briefly note why."
```

### Step 3: Evolution Narrative Generator

**New file:** `src/lib/ai/memory/narrative-generator.ts`

Produces the ticket evolution timeline by comparing consecutive analyses:

```typescript
// Input: Array of analysis pass results for a ticket (chronological)
// Output: Array of narrative entries
//
// For each consecutive pair of analyses:
// 1. What triggered the new analysis? (webhook event type)
// 2. What changed in situation_summary?
// 3. What changed in temperature?
// 4. What action items were added/completed/superseded?
//
// This can be done without LLM for structured changes (temperature, action items)
// Use LLM for summarizing situation diff into natural language
```

### Step 4: Timeline UI

**Modified file:** `src/components/dashboard/queues/support-action-board-view.tsx`

Add a "Timeline" tab or section in the expanded ticket view:

- Shows evolution entries in reverse chronological order
- Each entry: timestamp + trigger event + what changed
- Color-coded: green for improvements (temp down, actions completed), red for degradation (temp up, SLA risk)
- Collapsible for tickets with long histories

### Step 5: API Changes

**Modified file:** `src/app/api/queues/support-action-board/route.ts`

Add timeline data to the response. For performance, only include the last N entries (e.g., last 10 or last 24 hours):

```typescript

```

**New endpoint (optional):** `GET /api/queues/support-action-board/timeline/[ticketId]`
For full timeline history without loading it for every ticket.

### Step 6: Analysis Continuity Safeguards

Prevent the analysis from contradicting itself without reason:

1. **Temperature momentum:** If temperature went from calm → frustrated, the system shouldn't flip back to calm on the next analysis unless there's clear evidence of improvement (customer explicitly thanked, issue confirmed resolved). The review pass (Phase 5) should flag unjustified temperature reversals.

2. **Action item stability:** The action item pass should prefer keeping existing items over generating new ones unless the ticket state has genuinely changed. Phase 4's keep/supersede/create logic already supports this.

3. **Summary continuity:** The situation summary should build on the previous one, not rewrite it. "Previously: billing sync issue under investigation. Update: engineering fix deployed to staging" — not a completely new summary that ignores the history.

## Database Usage

This phase primarily uses the `analysis_pass_results` table created in Phase 2. No new tables needed. Additional indexes:

```sql
-- Efficient lookup of latest pass result per ticket per pass type
CREATE INDEX IF NOT EXISTS idx_pass_results_latest
  ON analysis_pass_results(hubspot_ticket_id, pass_type, created_at DESC);
```

## Testing Plan

1. **Change detection:** Analyze a ticket. Have someone reply. Run change detection. Verify it correctly identifies the new message and previous analysis.
2. **Memory-aware situation pass:** Analyze a ticket, then re-analyze after a change. Verify the situation summary references the change specifically, not just rewriting from scratch.
3. **Memory-aware action items:** Analyze a ticket with 3 actions. Complete one. Re-analyze. Verify the new analysis acknowledges the completion and doesn't re-suggest the completed action.
4. **Temperature continuity:** Analyze a ticket as "frustrated." Re-analyze with no significant changes. Verify it stays "frustrated" and doesn't randomly flip to "calm."
5. **Narrative timeline:** Run 3 analyses on a ticket (with changes between each). Verify the timeline shows 3 entries with correct change summaries.
6. **Timeline UI:** Open a ticket in the expanded view. Verify the timeline section renders correctly with chronological entries.
7. **First analysis:** Analyze a brand-new ticket. Verify it runs normally without errors about missing previous context.
